# “FCAJ Community Day 27/06” Event

## Purpose of the Event

The **FCAJ Community Day – June 2026** event was organized to create a space for the technology community to connect, share practical experience, and stay updated on emerging trends in **AI, Cloud, DevOps, Voice AI, security, and enterprise automation**.

For Information Technology students, the event provided a valuable opportunity to learn from speakers who are currently working in real-world enterprise environments. The sessions did not focus only on theory; they also included career stories, technical demonstrations, system architectures, and practical lessons from actual projects.

The main purposes of the event included:

* Connecting the technology community and creating an environment for networking and learning.
* Updating participants on new trends in AI, Cloud, Agentic Platforms, and Multi-Agent systems.
* Sharing experience in operating Cloud infrastructure within enterprises.
* Introducing practical applications of Voice AI, DevOps AI Agents, and Amazon Q.
* Providing career orientation for students in the AI era.
* Helping participants better understand the role of humans when working alongside AI.

Through the event, I gained a more practical perspective on how the IT industry is changing. AI is no longer only a tool that supports programming; it is also becoming involved in infrastructure operations, incident response, recruitment, customer service, and enterprise data security.

## Key Highlights

### Cloud Thinking – AI and Cloud Infrastructure Operations

The first session was presented by **Steve Trần**, Founder of Cloud Thinker. He shared his career journey from university, through his time working at Amazon Web Services, to founding Cloud Thinker to solve real-world Cloud infrastructure operations challenges.

One particularly noteworthy point was his perspective on the impact of AI on the labor market. AI will not completely replace humans, especially in critical fields such as Cloud infrastructure, security, and system operations. However, AI will change the requirements placed on engineers. In the future, engineers will need stronger skills, a deeper understanding of systems, and the ability to use AI effectively to solve complex problems.

The speaker also introduced Cloud Thinker’s **Agentic Platform**, which is designed specifically for Cloud environments. The platform aims to support tasks such as:

* Resolving system incidents more quickly.
* Supporting source code quality checks.
* Optimizing Cloud operating costs using a FinOps approach.
* Supporting security testing and penetration testing.
* Reducing the time required to investigate the root causes of system failures.

From this session, I understood that AI is gradually becoming an important part of infrastructure operations. However, AI can only be effective when it is provided with sufficient system context and is properly controlled by humans.

### Single-Agent and Multi-Agent Architectures in Enterprises

An important topic in the Cloud Thinking session was the difference between **Single-Agent** and **Multi-Agent** architectures.

With a **Single-Agent** approach, one Agent must handle many different tasks. This approach may be simple in the early stages, but it can become limited as the system grows because the Agent has to process too much context, access permissions become difficult to control, and optimization for individual task types becomes more challenging.

In contrast, a **Multi-Agent** architecture divides the system into multiple specialized Agents. Each Agent has a separate role, such as log analysis, security checking, cost optimization, code review, or incident handling. This approach helps the system:

* Optimize the context window.
* Reduce operating costs.
* Improve role-based access control.
* Handle specialized problems more effectively.
* Scale and remain maintainable as the system grows.

Through this section, I realized that Multi-Agent architecture is well suited to enterprise environments, where systems are often complex, contain large amounts of data, and have strict requirements for security, control, and explainability.

### Voice AI – Giving AI a Voice

The next session focused on **Voice AI**, with presentations by Hiếu Nghị, Anh Kiệt, and Trung Đào. This was a very interesting topic because Voice AI has significant development potential in Vietnam.

The speakers explained that Vietnamese voice data is still relatively limited. As a result, Vietnamese is considered a low-resource language in the field of speech AI. This makes building Voice AI systems for Vietnamese more challenging than for widely used languages such as English.

Two main Voice AI models were introduced:

* **Speech-to-Speech:** Converts input speech directly into output speech.
* **STT – LLM – TTS:** Converts speech into text, processes the text using a large language model, and then converts the resulting text back into speech.

For Vietnamese, the current practical solution is usually the **STT → LLM → TTS** model. This process allows the system to control content more effectively and reduce the risk of AI producing incorrect or unwanted responses.

The advanced features demonstrated included:

* A Voice Agent that provides consultation on Apple products.
* Speaker gender recognition.
* Natural handling of user interruptions.
* Collaboration between AI and humans in banking systems.
* Customer service applications in enterprises such as VPBank and VIB.

From this session, I understood that Voice AI is not only about speech conversion. It is a combination of audio processing, language models, Vietnamese-language data, user experience, and content control.

### DevOps AI Agents and Automated Incident Response

The session presented by Chị Bảo and Anh Nguyên Nguyễn from Cloud Kinetics focused on **DevOps AI Agents**. This solution helps automate the process of investigating system incidents.

In traditional DevOps, when an incident occurs, engineers often need to inspect many different data sources, such as logs, metrics, traces, dashboards, and system documentation. These data sources are often fragmented, difficult to correlate, and time-consuming to analyze when identifying the root cause.

Common problems in traditional DevOps include:

* Distributed log and tracing data.
* Knowledge gaps between technical teams.
* Long incident-resolution times.
* Difficulty identifying the root cause quickly.
* Heavy dependence on the individual experience of engineers.

The **DevOps AI Agent** solution uses Amazon Bedrock to support the investigation process. The Agent can learn the system context, analyze logs, propose hypotheses, search for evidence, and recommend possible solutions.

Its operating mechanism includes:

* **Context Learning:** The AI learns the system context, architecture, logs, and related data.
* **Hypothesis Investigation:** The AI proposes hypotheses about the possible causes of the incident.
* **Root Cause Analysis:** The AI identifies the root cause based on evidence.
* **Mitigation Plan:** The AI recommends a remediation plan.
* **Human-in-the-loop:** Humans review the results and make the final decision.

The demonstration of handling a DDoS attack against an e-commerce system helped me clearly see the practical value of AI in operations. Instead of spending several hours on manual investigation, a DevOps AI Agent can help reduce the investigation time to only a few dozen minutes.

### AI in Human Resources with Amazon Q / Amazon Quick

The session on **AI and human resources**, presented by Anh Trường and Chị Minh Anh, helped me understand another highly practical use of AI in enterprises: automating the recruitment process.

In traditional HR work, manually screening CVs is often time-consuming, may cause suitable candidates to be overlooked, and lacks a consistent evaluation framework. When the number of CVs is large, the recruitment process can become lengthy and negatively affect hiring quality.

The speakers introduced the use of **Amazon Q / Amazon Quick** to support recruitment activities, including:

* Creating job descriptions.
* Automatically screening CVs.
* Scoring CVs according to defined criteria.
* Comparing candidates with job requirements.
* Scheduling interviews.
* Sending emails to candidates.
* Connecting with internal systems such as Google Workspace or Microsoft 365.

A noteworthy point is that AI not only saves time but also helps standardize the evaluation process. As a result, enterprises can shorten recruitment time, improve candidate quality, and ensure that data is processed in a more secure environment.

From this session, I realized that AI is not limited to technical work. It can also support many other enterprise departments, including human resources, operations, customer service, and data management.

### Security and Technical Integration for Amazon Q

The final session focused on technical and security considerations when deploying Amazon Q in an enterprise. The speaker emphasized that when AI is connected to internal data, security must be treated as a mandatory priority.

One important topic was establishing a **Private Connection** for Amazon Q through a VPC, which helps reduce risks associated with transmitting data over the public internet. This is especially necessary for enterprises that handle sensitive data or have strict compliance requirements.

The session also discussed the **MCP Server – Model Context Protocol**. This method allows AI to connect to third-party systems such as Zalo, Gmail, WhatsApp, or internal applications. With MCP, AI can interact with real user data in a more controlled and secure manner.

Important deployment considerations include:

* Establishing private connectivity through a VPC.
* Controlling access permissions to data.
* Managing Route 53 Resolver costs.
* Calculating Application Load Balancer costs.
* Monitoring generated data traffic.
* Ensuring that AI can access only the data for which it has been granted permission.

From this session, I understood that deploying AI in an enterprise is not only about building a chatbot or an Agent. It also requires the design of a secure architecture, data controls, and operating-cost optimization.

## What I Learned

### The Importance of Context in AI Systems

One major lesson from the event is that AI can only operate effectively when it has sufficient context. In systems such as Cloud Operations, DevOps Agents, Voice AI, and Amazon Q, the AI needs to understand the data, objectives, access permissions, and system state in order to produce accurate results.

Without sufficient context, AI may provide incorrect answers, make unsupported inferences, or offer inappropriate recommendations. Therefore, when building an AI system, the design of input data, the knowledge base, and the context pipeline is extremely important.

### The Role of Humans in AI Systems

Across many sessions, I noticed that humans still play the decisive role in critical systems. AI can analyze information, make recommendations, and accelerate processing, but in areas such as banking, security, Cloud infrastructure, and recruitment, humans still need to verify the results and make the final decision.

The **Human-in-the-loop** model helps balance the speed of AI with human control. This is a safe and appropriate approach when applying AI in enterprise environments.

### Knowledge of Multi-Agent Systems and Agentic Platforms

The event helped me better understand the evolution from simple chatbots to Agentic Platforms. Instead of only answering questions, an AI Agent can perform tasks, retrieve data, analyze systems, and recommend actions.

Multi-Agent architecture divides work among multiple specialized Agents, thereby improving processing efficiency and making the system easier to control. This is an important direction for the development of enterprise AI.

### Knowledge of Voice AI for Vietnamese

I learned that building Voice AI for Vietnamese presents many challenges because Vietnamese-language data is still limited. The STT → LLM → TTS process is currently a suitable choice because it offers better content control and reduces the risk of AI generating inaccurate responses.

Voice AI can be applied to customer service, automated call centers, banking support, product consultation, and many other digital services.

### Knowledge of DevOps and AI-Assisted Incident Response

From the DevOps AI Agent session, I understood that incident response in large systems involves more than simply reading logs. Engineers must analyze multiple data sources. AI can help engineers shorten the investigation process, identify the root cause, and recommend possible solutions.

However, engineers still need to understand the system in order to verify whether the AI’s conclusions are correct. Therefore, AI does not replace DevOps Engineers; it helps them work more effectively.

### Knowledge of Security When Deploying AI

I learned that when AI is connected to enterprise data, security must be a top priority. Private VPC connections, permission mechanisms, data controls, and the MCP protocol are important elements for deploying AI safely.

This helped me understand that an enterprise AI system must satisfy three factors: effectiveness, security, and controllability.

## Application to Work and Study

The knowledge gained from the event can be applied directly to my studies, projects, and career development.

First, I can apply **Context Engineering** when using AI for learning. When asking AI to help write a report, analyze code, or explain a concept, I should provide a clear objective, context, requirements, and related data.

Second, I can use AI tools such as Amazon Q or Agentic Platforms to support coding, document review, error analysis, and infrastructure deployment for academic projects.

Third, I realized that CVs may be screened by AI in the future. Therefore, I need to present my CV more clearly, emphasize practical skills, personal projects, technologies used, and measurable outcomes.

Fourth, I will focus more on systems thinking instead of learning only how to code. A good engineer needs to understand how systems operate, how incidents are handled, how data is protected, and how Cloud costs are optimized.

Fifth, I can apply knowledge of DevOps AI Agents to personal projects by building logging, metrics, and alerting systems, and then experimenting with AI tools that support incident analysis.

Finally, the knowledge of AI security makes me more cautious when connecting AI to personal or enterprise data. AI should not be granted access to sensitive data without appropriate permission and control mechanisms.
